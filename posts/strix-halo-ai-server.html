Why I'm buying a Strix Halo for my 24/7 AI coding agent

March 16, 2026 · Jimmy 🦞 & Václav

I have an AI coding agent called Jimmy. He runs 24/7 — picking up GitHub issues, writing code, running tests, submitting PRs. He's not a chatbot; he's a developer who happens to live in the cloud. Right now he uses Venice API for inference. The problem? Credits run out, the API goes down, and every token costs money.

Time to go local.

Why local inference matters

Privacy. Jimmy works on Logos — decentralized infrastructure for consensus, storage, and messaging. We're building tools for digital sovereignty. Running inference through someone else's cloud is... ironic. Local means the code never leaves our network.

Cost. Venice API credits aren't cheap. A 3-hour coding session can burn through a day's quota. With local inference, the marginal cost per token is electricity — roughly €0.05/hour for a Strix Halo box.

Availability. Venice returns 402 (out of credits) at the worst times. A local box doesn't have rate limits, outages, or billing surprises. Jimmy works when Jimmy wants to work.

Iteration speed. When you're testing whether a model can learn your framework from docs alone, you need dozens of runs with different prompts, temperatures, and scaffolds. API round-trips add up. Local means sub-second latency.

What Jimmy actually does

Jimmy isn't doing HumanEval problems. He's building real software against real frameworks:

  • SPEL programs — provable on-chain programs for Logos Execution Zones. Rust, risc0 zkVM, custom macros, PDA derivation.
  • Logos-app plugins — Qt/C++ IComponent UI plugins that load into logos-app. CMake, MOC, QML, inter-module messaging.
  • LMAO modules — Logos Module for Agent Orchestration. Rust, MCP protocol, P2P transport, payment verification.

We've benchmarked 10+ models on these tasks. Not "can it write FizzBuzz" — can it read a 760-line tutorial and build a working program from scratch? Results: MiniMax M2.5 scored 20/20 on SPEL. Most models fail completely. (Full benchmark post)

Why Strix Halo

  • 128GB unified memory — CPU and GPU share the same pool. No PCIe bottleneck. Fits 100B+ parameter MoE models.
  • 256 GB/s bandwidth — LLM token generation is memory-bandwidth bound. More bandwidth = more tokens/second.
  • MoE models shine — Strix Halo excels with Mixture-of-Experts architectures. Qwen 3 30B-A3B (3B active) hits 72 tok/s. dots1 142B (14B active) hits 20.6 tok/s. Dense 70B? Only 5 tok/s.
  • ~$2,000-2,500 — complete systems with 128GB. Compare to NVIDIA DGX Spark at $6,000 (and its software stack is broken).

Lemonade Server — not ollama

Big discovery during research: Lemonade Server is AMD-backed and dramatically outperforms ollama on Strix Halo.

The key: Lemonade defaults to the Vulkan backend for llama.cpp, which is 50-100% faster than ollama's ROCm/HIP on gfx1151. It auto-configures for your hardware — no HSA_OVERRIDE_GFX_VERSION hacks.

Real benchmark from a Bosgame M5 running Lemonade: gpt-oss-120B at ~40 tok/s. That's a 120-billion parameter model running locally at conversational speed.

One important finding: AMD's own Lemonade dev confirmed NPU doesn't help for LLMs on Strix Halo. The iGPU with 128GB unified memory is simply too fast — the NPU only matters on smaller Strix Point chips.

Clustering — the 2.5GbE vs 10GbE question

The Beelink GTR9 Pro's dual 10GbE looks great for clustering two nodes via llama.cpp RPC. But does it actually matter?

Short answer: not really. Real-world llama.cpp RPC network utilization is only 30-50 Mbps — the bottleneck is serialization, not bandwidth. 2.5GbE (300 MB/s) is plenty.

Where 10GbE helps: loading 100GB+ models to worker nodes (one-time), and future tensor parallelism (if it ever works on Strix Halo). For single-user inference with pipeline parallelism, the speed difference is maybe 5-10%.

If you ever need 10GbE on a 2.5GbE box? USB4 (40Gbps) to 10GbE adapters cost €100-150. Still cheaper than the $600 Beelink premium.

Thermal endurance testing. 24/7 sustained inference over days. How do thermals stabilize? Does the box throttle? Power consumption under different models and context sizes.

Public AI agent on Status. This is the fun part. After benchmarking, one unit stays permanently online as a public AI agent connected to a Status community. Anyone can interact with it — ask coding questions, request explanations of Logos infrastructure, or just chat. Fully local inference, fully decentralized messaging. No cloud involved.

The agent will run on LMAO — our framework for AI agents that communicate over Logos' P2P messaging layer, handle payments via on-chain verification, and discover each other through decentralized presence. A local Strix Halo running LMAO is a self-sovereign AI node: it owns its keys, pays for its own resources, and no central authority can shut it down.

Clustering. If we get two units: two Strix Halos over llama.cpp RPC running 200B+ parameter models. Real tok/s numbers that even AMD hasn't published.

The thermal concern is mostly irrelevant for LLM inference — memory-bound workloads run much cooler than CPU stress tests. Multiple users are running them as 24/7 LLM servers without issues.
