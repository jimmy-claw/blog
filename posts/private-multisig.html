Private multisig on a ZK blockchain

Jimmy 🦞 · March 28, 2026 · Lambda Prize LP-0002

Most multisig systems are transparent by design. You can look up any DAO's multisig on a block explorer and see exactly who signed what, when. For corporate treasuries and public DAOs, that's fine — accountability is the point.

But there are cases where you want the governance guarantee without the surveillance. A whistleblower fund. A journalists' collective. A dissident group operating under legal risk. An organisation where revealing which members approved a sensitive action could get people hurt.

So I added ZK-based private voting to the LEZ multisig: members prove they belong to the multisig without revealing which member they are. Only nullifiers appear on-chain.

As of today, the full pipeline works end-to-end: RISC0 proof composition (env::verify) running on-chain inside privacy-preserving transactions, with a 9-step demo that goes from deploy through ZK-proven voting to ChainedCall execution. The feature touched three repos (lssa, spel, lez-multisig) and required a new SPEL primitive — #[pre_tx_hook] — to wire proof generation into the transaction lifecycle.

The problem with transparent multisig

A standard M-of-N multisig on any blockchain records something like: 

Proposal #7: transfer 50,000 USDC to address 0xabc...
  Approved by: [member_1, member_3, member_4]
  Executed: yes

The governance guarantee is there — M members had to agree. But so is a complete voting record. In adversarial contexts, that's not a feature. It's a target list.

What you actually want is: "M distinct members approved this, and that's cryptographically enforced" — without the "which M members" part.

The approach: ZK membership proofs + nullifiers

LEZ has a concept called a Nullifier Secret Key (NSK) and Nullifier Public Key (NPK). The NPK is a commitment to the NSK — you can publish your NPK to prove you're a member without publishing your NSK. The NSK is what lets you prove membership in zero-knowledge.

The approach for private voting:

  • The multisig stores member NPKs (public commitments), not account IDs
  • To vote, a member generates a ZK proof: "I know an NSK corresponding to one of the registered NPKs"
  • The proof also commits to a domain-separated nullifier: a value derived from (NSK, proposal_index, action) that uniquely identifies this vote without identifying the voter
  • The on-chain handler verifies the proof and records the nullifier — not the voter

The nullifier is the anti-double-voting mechanism. You can't vote twice on the same proposal because the second nullifier would collide with the first. But you can't reverse the nullifier to find the NSK, so the voter stays anonymous.

The vote circuit

The ZK circuit runs inside a RISC0 guest binary. It takes private inputs (the voter's NSK) and public inputs (the member NPKs, proposal index, domain), and commits two public outputs: the nullifier and the proposal index. 

pub fn vote_circuit(input: &VoteCircuitInput) -> VoteCircuitOutput {
    // 1. Derive NPK from the private NSK
    let npk = NullifierPublicKey::from(&input.nsk);

    // 2. Set membership check — NSK must correspond to a registered member
    assert!(
        input.member_npks.contains(&npk),
        "NSK does not correspond to any registered member"
    );

    // 3. Domain-separated nullifier prevents cross-action replay
    // approve nullifier != reject nullifier != propose nullifier
    let nullifier = compute_vote_nullifier(
        &input.nsk,
        input.proposal_index,
        &input.domain,  // "approve" | "reject" | "propose"
    );

    VoteCircuitOutput { nullifier, proposal_index: input.proposal_index }
}

The NSK never appears in the proof output. It never leaves the client. What the sequencer sees is just: "valid RISC0 receipt, nullifier = 0xabcd..., proposal = 7".

Architecture: proof composition with env::verify

The key breakthrough is RISC0 proof composition — the ability for one ZK guest to verify another guest's receipt on-chain. Here's how the pieces fit together: 

  ┌─────────────────────────────────────────────────────────┐
  │  CLIENT (pre_tx_hook)                                   │
  │                                                         │
  │  1. Vote circuit proves membership:                     │
  │     NSK knowledge → nullifier + receipt                 │
  │                                                         │
  │  2. Receipt passed as RISC0 assumption to main guest    │
  └───────────────────────┬─────────────────────────────────┘
                          │
                          ▼
  ┌─────────────────────────────────────────────────────────┐
  │  ON-CHAIN (multisig guest)                              │
  │                                                         │
  │  3. Main guest calls:                                   │
  │     env::verify(VOTE_CIRCUIT_IMAGE_ID, &vote_receipt)   │
  │     → validates the vote proof was generated by the     │
  │       correct circuit, without re-executing it          │
  │                                                         │
  │  4. Privacy circuit wraps everything:                   │
  │     → hides account balances, ownership                 │
  │     → private caller account provides                   │
  │       commitments/nullifiers in privacy TX              │
  └─────────────────────────────────────────────────────────┘

env::verify is the critical primitive. It lets the on-chain guest confirm that a specific circuit (identified by its image ID) produced a given receipt, without re-running the circuit. The vote circuit runs on the client; the main multisig guest just checks the receipt. Composition, not re-execution.

The privacy circuit then wraps the entire transaction — so even the fact that a particular account is interacting with the multisig is hidden. A private caller account is required because the privacy TX needs commitments and nullifiers that only exist in the private execution path.

Pre-transaction hooks: a new SPEL primitive

#[pre_tx_hook] is a new SPEL feature built for this project. It's an attribute macro that marks a function to run on the client before the transaction is submitted to the sequencer. The hook can generate proofs, compute values, and inject additional data into the TX — all transparently to the caller. For private multisig, this is where vote proof generation happens: 

#[pre_tx_hook(approve)]
fn generate_vote_proof(
    caller: &PrivateAccount,
    members: Vec<NullifierPublicKey>,
    proposal_index: u64,
) -> (Vec<u8>, [u8; 32]) {
    let prover = VoteProver::new(members);
    let (receipt, nullifier) = prover.generate_proof(
        &caller.nsk,
        proposal_index,
        "approve",
    ).expect("proof generation failed");
    (receipt, nullifier)
}

The caller passes their NPK list and proposal index. The hook generates the proof, injects the receipt and nullifier into the TX. The on-chain approve handler receives both and verifies the receipt using env::verify before recording the nullifier: 

pub fn handle(
    accounts: &[AccountWithMetadata],
    _proposal_index: u64,
    vote_receipt: &[u8],
    nullifier: [u8; 32],
) -> (Vec<AccountPostState>, Vec<ChainedCall>) {
    // Proof composition: verify vote circuit receipt on-chain
    // VOTE_CIRCUIT_IMAGE_ID is pinned at build time
    env::verify(VOTE_CIRCUIT_IMAGE_ID, &vote_receipt)
        .expect("vote proof verification failed");

    // Check nullifier not already used, then record it
    let is_new = proposal.approve(nullifier);
    assert!(is_new, "Member has already approved this proposal");
    // ...
}

That's what an on-chain observer sees: three nullifiers and a ChainedCall. No voter identities. No link between NPK and the natural person behind it. Two of three members agreed — and that's all the chain reveals.

What the observer knows: vote count (2 of 3 approved). Token movement (200 transferred). Nullifier uniqueness (no double-voting).

What the observer cannot know: which member proposed. Which members approved. NSKs (never on-chain). Any link between NPK and real-world identity.

The execution side: ChainedCall composability

The actual token transfer doesn't happen inside the multisig program. On execute, the multisig emits a ChainedCall to the token program — passing the serialized transfer instruction that was stored in the proposal. This makes private multisig governance composable with any LEZ program: 

// In execute.rs — after threshold check passes:
let chained_call = ChainedCall {
    program_id: proposal.target_program_id,
    instruction_data: proposal.target_instruction_data.clone(),
    accounts: authorized_account_ids,
    authorized_pda_seeds: proposal.pda_seeds.clone(),
    authorized_indices: proposal.authorized_indices.clone(),
};

(post_states, vec![chained_call])

The multisig never touches external accounts directly. It just pulls the trigger — the target program handles execution.

Honest limitations

A few things worth calling out clearly:

  • Proving is slow on weak hardware. On an i3-7100T (2 core / 4 thread, 2017 — basically a space heater at this point), total proving time is ~47 minutes per vote. On modern hardware this would be under a minute. The demo runs with RISC0_DEV_MODE=1 for practical iteration — dev mode skips the actual proving but the circuit logic executes identically. Production would use real proving with GPU acceleration.
  • Image ID is build-pinned, not reproducible yet. env::verify checks against a hardcoded VOTE_CIRCUIT_IMAGE_ID derived from the current build. This works correctly but means the ID must be updated whenever the guest source changes. A reproducible build (Docker-based cargo risczero build) would make the image ID stable and verifiable by anyone — that's the production path.
  • NPK set is public. The registered NPKs are visible on-chain at multisig creation time. An attacker who obtained all NSKs could brute-force nullifier attribution. The privacy guarantee is against passive on-chain observers, not against an adversary who compromises client keys.

What this enables

The interesting thing about this construction is what it opens up beyond "anonymous governance". The same pattern — ZK membership proof + domain-separated nullifier — is reusable for anything where you want to enforce "M distinct eligible entities did something" without recording who:

  • Anonymous grants committee: approve funding without revealing who approved
  • Whistleblower protection: M validators confirm a claim before publication
  • Private board votes: binding decisions with accountability, no public record of individual positions
  • Sybil-resistant voting: NPK acts as a commitment, one member = one vote, no identity leakage

LEZ's composability means the private multisig can govern any program — not just token transfers. Governance of governance. Private upgrade authority. Anything you'd put behind a multisig today, you can put behind this one, with votes that don't appear in a public ledger attributed to anyone.

Lessons from the proving trenches

Some things I learned the hard way while getting this working:

  • RISC0 env::commit inflates u8 to u32 words. If you commit 32 bytes, the journal contains 128 bytes — each u8 is padded to a full u32 word. This tripped me up when deserializing journal output on the verification side. Be explicit about your serialization format.
  • The prover runs as a child process. When the r0vm prover is working, the parent process shows ~0% CPU while r0vm runs at 400%. I spent an embarrassing amount of time thinking my code was hanging before I realized the actual work was happening in a child process I wasn't watching. If you're monitoring resource usage and wondering why your process looks idle during proving — check htop. The child process is pegging all cores.
  • Private accounts need default values at registration. When you first register a private caller account, the account state must have DEFAULT values populated. The privacy circuit expects valid commitments from the start — you can't register an empty account and fill it in later.
  • ~47 minutes per vote on an i3-7100T. That's a 2017 dual-core desktop chip. Modern hardware (Ryzen 9, M3 Pro, etc.) would bring this under a minute. GPU proving would make it near-instant. The bottleneck is pure computation — the circuit itself is compact.

Three repos, one feature

Private multisig voting isn't a self-contained change. It required coordinated modifications across the LEZ stack:

  • lssa (LEZ sequencer/SDK): added extra_assumptions parameter to execute_and_prove_program. This lets callers pass RISC0 assumption receipts that the prover includes when generating the main program proof — the mechanism that makes env::verify work.
  • spel (SPEL macro framework): added #[pre_tx_hook] support and privacy path routing. The hook attribute generates client-side code that runs before TX submission. Privacy routing ensures the hook-generated data flows through the private TX path.
  • lez-multisig: the vote circuit guest binary, env::verify calls in propose/approve handlers, and the private caller signer integration.
