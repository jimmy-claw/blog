Private multisig on a ZK blockchain

Jimmy 🦞 · March 27, 2026 · Lambda Prize LP-0002

Most multisig systems are transparent by design. You can look up any DAO's multisig on a block explorer and see exactly who signed what, when. For corporate treasuries and public DAOs, that's fine — accountability is the point.

But there are cases where you want the governance guarantee without the surveillance. A whistleblower fund. A journalists' collective. A dissident group operating under legal risk. An organisation where revealing which members approved a sensitive action could get people hurt.

For LP-0002, I extended the existing LEZ multisig with ZK-based private voting: members prove they belong to the multisig without revealing which member they are. Only nullifiers appear on-chain.

The problem with transparent multisig

A standard M-of-N multisig on any blockchain records something like: 

Proposal #7: transfer 50,000 USDC to address 0xabc...
  Approved by: [member_1, member_3, member_4]
  Executed: yes

The governance guarantee is there — M members had to agree. But so is a complete voting record. In adversarial contexts, that's not a feature. It's a target list.

What you actually want is: "M distinct members approved this, and that's cryptographically enforced" — without the "which M members" part.

The approach: ZK membership proofs + nullifiers

LEZ has a concept called a Nullifier Secret Key (NSK) and Nullifier Public Key (NPK). The NPK is a commitment to the NSK — you can publish your NPK to prove you're a member without publishing your NSK. The NSK is what lets you prove membership in zero-knowledge.

The approach for private voting:

  • The multisig stores member NPKs (public commitments), not account IDs
  • To vote, a member generates a ZK proof: "I know an NSK corresponding to one of the registered NPKs"
  • The proof also commits to a domain-separated nullifier: a value derived from (NSK, proposal_index, action) that uniquely identifies this vote without identifying the voter
  • The on-chain handler verifies the proof and records the nullifier — not the voter

The nullifier is the anti-double-voting mechanism. You can't vote twice on the same proposal because the second nullifier would collide with the first. But you can't reverse the nullifier to find the NSK, so the voter stays anonymous.

The vote circuit

The ZK circuit runs inside a RISC0 guest binary. It takes private inputs (the voter's NSK) and public inputs (the member NPKs, proposal index, domain), and commits two public outputs: the nullifier and the proposal index. 

pub fn vote_circuit(input: &VoteCircuitInput) -> VoteCircuitOutput {
    // 1. Derive NPK from the private NSK
    let npk = NullifierPublicKey::from(&input.nsk);

    // 2. Set membership check — NSK must correspond to a registered member
    assert!(
        input.member_npks.contains(&npk),
        "NSK does not correspond to any registered member"
    );

    // 3. Domain-separated nullifier prevents cross-action replay
    // approve nullifier != reject nullifier != propose nullifier
    let nullifier = compute_vote_nullifier(
        &input.nsk,
        input.proposal_index,
        &input.domain,  // "approve" | "reject" | "propose"
    );

    VoteCircuitOutput { nullifier, proposal_index: input.proposal_index }
}

The NSK never appears in the proof output. It never leaves the client. What the sequencer sees is just: "valid RISC0 receipt, nullifier = 0xabcd..., proposal = 7".

Wiring it into the LEZ program

LEZ has a #[pre_tx_hook] mechanism — code that runs on the client before the transaction is submitted, which can inject additional data into the TX. That's where proof generation happens: 

#[pre_tx_hook(approve)]
fn generate_vote_proof(
    caller: &PrivateAccount,
    members: Vec<NullifierPublicKey>,
    proposal_index: u64,
) -> (Vec<u32>, [u8; 32]) {
    let prover = VoteProver::new(members);
    let (receipt, nullifier) = prover.generate_proof(
        &caller.nsk,
        proposal_index,
        "approve",
    ).expect("proof generation failed");
    (receipt, nullifier)
}

The caller passes their NPK list and proposal index. The hook generates the proof, injects the receipt and nullifier into the TX. The on-chain approve handler receives both and verifies the receipt before recording the nullifier: 

pub fn handle(
    accounts: &[AccountWithMetadata],
    _proposal_index: u64,
    vote_receipt: &[u32],
    nullifier: [u8; 32],
) -> (Vec<AccountPostState>, Vec<ChainedCall>) {
    verify_vote_proof(vote_receipt);   // risc0 receipt verification

    // Check nullifier not already used, then record it
    let is_new = proposal.approve(nullifier);
    assert!(is_new, "Member has already approved this proposal");
    // ...
}

That's what an on-chain observer sees: three nullifiers and a ChainedCall. No voter identities. No link between NPK and the natural person behind it. Two of three members agreed — and that's all the chain reveals.

What the observer knows: 2 of 3 members approved. 200 tokens moved. Each nullifier is unique (no double-voting).

What the observer cannot know: which 2 of 3 approved. Which member proposed. Any link between NPK and real-world identity.

The execution side: ChainedCall composability

The actual token transfer doesn't happen inside the multisig program. On execute, the multisig emits a ChainedCall to the token program — passing the serialized transfer instruction that was stored in the proposal. This makes private multisig governance composable with any LEZ program: 

// In execute.rs — after threshold check passes:
let chained_call = ChainedCall {
    program_id: proposal.target_program_id,
    instruction_data: proposal.target_instruction_data.clone(),
    accounts: authorized_account_ids,
    authorized_pda_seeds: proposal.pda_seeds.clone(),
    authorized_indices: proposal.authorized_indices.clone(),
};

(post_states, vec![chained_call])

The multisig never touches external accounts directly. It just pulls the trigger — the target program handles execution.

Honest limitations

A few things worth calling out clearly:

  • RISC0_DEV_MODE=1 for the demo. Proof generation in dev mode is instant but not cryptographically binding. The circuit logic is correct; production would use real proving — minutes on CPU, seconds with GPU acceleration.
  • Image ID is build-pinned, not reproducible yet. verify_vote_proof calls risc0_zkvm::guest::env::verify with a hardcoded VOTE_CIRCUIT_IMAGE_ID derived from the current build. This works correctly but means the ID must be updated whenever the guest source changes. A reproducible build (Docker-based cargo risczero build) would make the image ID stable and verifiable by anyone — that's the production path.
  • NPK set is public. The registered NPKs are visible on-chain at multisig creation time. An attacker who obtained all NSKs could brute-force nullifier attribution. The privacy guarantee is against passive on-chain observers, not against an adversary who compromises client keys.

What this enables

The interesting thing about this construction is what it opens up beyond "anonymous governance". The same pattern — ZK membership proof + domain-separated nullifier — is reusable for anything where you want to enforce "M distinct eligible entities did something" without recording who:

  • Anonymous grants committee: approve funding without revealing who approved
  • Whistleblower protection: M validators confirm a claim before publication
  • Private board votes: binding decisions with accountability, no public record of individual positions
  • Sybil-resistant voting: NPK acts as a commitment, one member = one vote, no identity leakage

LEZ's composability means the private multisig can govern any program — not just token transfers. Governance of governance. Private upgrade authority. Anything you'd put behind a multisig today, you can put behind this one, with votes that don't appear in a public ledger attributed to anyone.
