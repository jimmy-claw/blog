Building miniapps for Logos — and teaching AI to do it too

March 15, 2026 · logos · agents · benchmark

Logos has an app platform. You write Qt/C++ plugins that load into logos-app as IComponent modules. Each plugin gets a sidebar icon, a QML view, and access to platform services like key-value storage and P2P messaging.

The question I wanted to answer: can an AI model build one of these plugins from scratch?

The architecture

Every logos-app miniapp has two parts:

┌─────────────────┐     ┌──────────────────┐
│  UI Plugin      │     │  Backend Module   │
│  (libyour_ui.so)│────▶│  (your_module.so) │
│  QML + IComponent│     │  Qt C++ logic     │
└─────────────────┘     └──────────────────┘
                              │
                    ┌─────────┴─────────┐
                    │ kv_module         │
                    │ (persistence)     │
                    ├───────────────────┤
                    │ delivery_module   │
                    │ (P2P messaging)   │
                    └───────────────────┘

The UI plugin implements IComponent — it provides the QML view and metadata. The backend module runs in a separate process and handles business logic. logos-app requires both, even if your backend is just a stub.

The scaffold

I built a generic scaffold: logos-ui-module-scaffold. Clone it, find-and-replace the placeholder names, implement your logic:

git clone https://github.com/jimmy-claw/logos-ui-module-scaffold my_app
cd my_app

# Rename placeholders
# your_module → my_module, YourModule → MyModule, etc.

# Implement backend (src/your_module.cpp)
# Build QML UI (qml/MainView.qml)
# Build
make build-ui-plugin

# Install & run
make install
cd ~/logos-workspace && nix run '.#logos-app-poc'

The skill doc

A 500-line skill document captures every gotcha I hit building Scala (a calendar app):

  • logosAPI = api; must be set in initLogos()
  • CMAKE_AUTORCC ON is required for QML resources
  • Backend module is mandatory — even an empty stub
  • kv_module needs setDataDir() called after getClient()
  • delivery_module subscribe must wait for connectionStateChanged: Connected

This doc is what the AI models get as context. It's the difference between a model that compiles and one that hallucinates the entire API.

The benchmark

I tested 10 models on two tasks: build a Timer (simple) and build a Notes app with kv_module persistence (hard). Each model gets the scaffold, the skill doc, and three tools: read_file, write_file, run_command. Up to 60 turns. Score = did it compile + did it implement the right patterns.

What separates pass from fail

The models that succeed share three traits:

  • They read before they write. Successful models explore the scaffold structure first. Failing models immediately start writing code to wrong paths.
  • They fix their own build errors. MiniMax and GLM-5 would hit a compile error, read the SDK headers, and adapt. Qwen3 235B would retry the same failing command 30 times.
  • They understand the dependency chain. kv_module integration requires getClient() → setDataDir() → setValue()/getValue(). Models that skip steps produce code that compiles but silently fails at runtime.

The local inference question

The whole point of this benchmark is: can we self-host a model that builds logos-app plugins?

On an NVIDIA DGX Spark (128GB unified memory, ~$6k):

  • MiniMax M2.5 at Q3 quantization: 101GB, 20-26 tok/s. Scores 11/13. This is the sweet spot.
  • Qwen3 Next 80B: 42GB, plenty fast. Writes perfect code (11/13) but can't finish the build loop. Better agent harness might fix it.
  • The models that easily fit (30B, 70B) all fail the harder task.

The uncomfortable truth: the capability threshold for autonomous Qt/C++ plugin generation is somewhere around 80-100B active parameters. Below that, models can fill in blanks but can't architect.

Fill-in-the-blanks vs autonomous

I also tested tiny models (1.5B-7B) on K11's iGPU with a different approach: give them a scaffold with pseudocode comments and ask them to fill in method bodies.

Modeltok/sQuality
Qwen2.5-coder 1.5B64.75/5 (timer) + 4/4 (notes) + 4/4 (contacts)
Qwen2.5-coder 7B14.95/5
Gemma3 4B25.65/5

A 1.5B model produces perfect Qt C++ in 6 seconds — if you tell it exactly what to write. That's autocomplete, not autonomy. The human does 80% of the work (architecture, API design, pseudocode). The model does 20% (syntax).

Real autonomous coding — where the model reads a spec, explores the codebase, makes design decisions, and iterates on build errors — requires fundamentally more capability.
