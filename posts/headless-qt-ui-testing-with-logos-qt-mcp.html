Headless Qt UI Testing with logos-qt-mcp

March 24, 2026 · testing · qt · logos · agents

How do you test a Qt/QML plugin when there's no screen? You're building a Logos App module, it compiles fine, the QML looks right — but does it actually render? Does clicking the button do what you think it does?

On a headless CI box, you can't just look at it. And xdotool clicking at pixel coordinates is fragile garbage. We needed something better.

The problem

Logos App plugins are Qt/QML widgets loaded at runtime. Testing them means:

  • Launching the full app (plugins need the host to load)
  • Navigating to your plugin in the sidebar
  • Verifying that specific text/elements appear
  • Clicking buttons and checking results
  • Doing all this without a monitor

Screenshot-and-pray doesn't scale. We needed structured access to the QML tree.

The solution: logos-qt-mcp

logos-qt-mcp embeds an inspector directly into the Qt app. A tiny TCP server exposes the live QML object tree, and an MCP bridge makes it accessible to AI agents and test runners.

┌─────────────────┐ TCP/JSON ┌──────────────┐ stdio/MCP ┌─────────────┐ │ LogosApp + │◄───────────►│ MCP Server │◄───────────►│ Claude / │ │ InspectorServer │ port 3768 │ index.mjs │ │ Test Runner │ └─────────────────┘ └──────────────┘ └─────────────┘

The InspectorServer is a C++ class that attaches to your main window. It walks the QML tree, finds interactive elements, takes screenshots, and executes clicks — all over a TCP socket on port 3768.

The MCP server translates those raw TCP/JSON messages into proper MCP tool calls. So Claude (or any MCP client) can inspect your app, click around, and verify what it sees.

How to enable it

If you're building from logos-basecamp, the inspector is already wired up:

nix build .#x86_64-linux.appDistributedWithInspector -o result-app

For your own Qt app, add the cmake flag and attach the inspector:

cmake .. -DENABLE_QML_INSPECTOR=ON \
         -DLOGOS_QT_MCP_ROOT=$(pwd)/logos-qt-mcp
// In main.cpp, after creating your window:
#include "InspectorServer.h"
InspectorServer::attach(mainWindow);

MCP tools

Once the inspector is running, the MCP server exposes these tools:

  • qml_get_tree — full QML object tree with types, properties, geometry
  • qml_screenshot — PNG screenshot of the app
  • qml_click — click by text, objectName, or coordinates
  • qml_evaluate — run arbitrary JS in the QML context
  • qml_list_interactive — list all clickable elements
  • qml_find_and_click — find element by text and click it

The difference from xdotool: these operate on the actual QML tree. You click "Submit", not pixel (423, 187). If the layout shifts, your tests still work.

Writing tests

logos-qt-mcp ships a test framework. Build it once:

cd ~/logos-basecamp
nix build .#x86_64-linux.logosQtMcp -o result-mcp

Then write tests:

import { test, run } from "./result-mcp/test-framework/framework.mjs";

test("counter: increment works", async (app) => {
  await app.click("counter");                // open the counter plugin
  await app.expectTexts(["Count: 0"]);        // verify initial state

  await app.click("Increment");               // click the button
  await app.expectTexts(["Count: 1"]);        // verify it changed

  await app.screenshot();                     // save proof
});

run();

The framework handles MCP connection, error reporting, and cleanup. You just write assertions.

The CI pipeline

This is the full headless setup — no display, no GPU, just Xvfb and vibes:

#!/bin/bash
set -e

# Build everything
nix build .#x86_64-linux.logosQtMcp -o result-mcp
nix build .#x86_64-linux.appDistributedWithInspector -o result-app

# Virtual display
Xvfb :99 -screen 0 1920x1080x24 &
sleep 2

# Launch app
DISPLAY=:99 ./result-app/bin/logos-basecamp &
sleep 10

# Start MCP server
node result-mcp/mcp-server/index.mjs &

# Run tests
node tests/ui-tests.mjs

# Cleanup
kill %1 %2 %3

The module scaffold ships with tests/ci-test.sh that does exactly this — build, launch, test, cleanup. One command.

The agent angle

Here's where it gets interesting. Because the inspector speaks MCP, you can point Claude at a running app and say "test this plugin." Claude calls qml_get_tree to understand the UI, qml_click to interact, qml_screenshot to verify. It writes the tests and runs them.

The loop: agent builds plugin → app loads it → agent inspects it → agent verifies it works → agent fixes what's broken.

No human needed to eyeball the screen. The agent can see the QML tree directly.
